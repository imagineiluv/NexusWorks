1. 개요 본 설계서는 현재 배포된 파일 세트와 이번 패치 파일 세트를 비교하여, 운영 관점에서 필요한 검수 항목을 표준화하고 재사용 가능 한 범용 시스템으로 구현하기 위한 상세설계를 정리한 문서이다. 핵심 목표는 필수 파일 누락 탐지, JAR/XML 중심의 구조 비교, 운영팀과 개발팀이 함께 이해할 수 있는 시각화 리포트 제공이다. 핵심 설계 원칙 비교 기준을 코드에 고정하지 않고 기준 문서(Baseline)로 관리하여 프로젝트가 달라져도 반복 사용할 수 있게 설계 한다. 2. 범위 및 전제 • 비교 입력은 배포본 루트 경로, 패치본 루트 경로, 기준 문서 경로를 기본으로 한다. • 상대경로를 기준 키로 사용하여 운영본과 패치본 파일 목록을 병합한다. • JAR 은 해시 비교와 내부 엔트리 비교를 기본으로 하고, 필요 시 상세 비교를 확장한다. • XML 은 정규화 후 구조 비교를 기본으로 하며 주요 변경 XPath 를 기록한다. • 최종 결과는 HTML 대시보드와 Excel 리포트를 동시에 생성한다. 3. 상위 아키텍처 순서 모듈 역할 1 기준 문서 로더 Baseline 규칙을 읽고 필수 여부, 비교 방식, 제외 정책, 우선순위를 적재 2 파일 인벤토리 수집기 운영본과 패치본을 재귀 스캔하여 상대경로, 크기, 수정일시, 해시 생성 3 룰 매핑 엔진 정확 경로, 패턴, 확장자 기본 룰 순으로 파일 별 적용 정책 결정 4 유형별 비교 엔진 공통 비교, JAR 비교, XML 비교, 기타 Binary/Text 비교 수행 5 상태/위험도 판정 엔진 OK, CHANGED, ADDED, REMOVED, MISSING_REQUIRED 등 최종 상태 부여 6 집계/저장 모듈 비교 결과와 상세 diff 정보를 구조화하여 리 포트 생성용 데이터셋 구성 7 리포트/대시보드 엔진 HTML 대시보드, Excel 리포트, 필터링 목 록, 상세 팝업 데이터 출력 4. 기준 문서(Baseline) 설계 범용성을 확보하기 위해 비교 대상과 정책은 모두 기준 문서로 관리한다. 권장 형식은 baseline.xlsx 이며, 최소 RULES 시트를 포 함한다.

5. 내부 데이터 모델 모델 주요 필드 설명 FileInventory relative_path, current_exists, patch_exists, hash 운영/패치 파일의 기본 메타데이터 CompareResult status, severity, compare_mode, changed_summary 최종 상태 및 화면 출력용 요약 JarDetail added_entries, removed_entries, manifest_changed JAR 상세 비교 결과 XmlDetail changed_xpaths, changed_node_count XML 구조 변경 상세 정보 6. 처리 흐름 1) 비교 실행 요청 수신 2) 기준 문서 로딩 및 유효성 검증 3) 운영본/패치본 파일 스캔 4) 상대경로 기준 병합 5) 파일별 규칙 매핑 6) 유형별 비교 수행 7) 상태/위험도 판정 8) 집계 및 리포트 생성 7. 상태 및 위험도 판정 상태 위험도 예시 판정 기준 OK LOW 양쪽 존재하고 비교 결과 동일 CHANGED MEDIUM/HIGH 양쪽 존재하나 내용 또는 구조가 상이 ADDED LOW 패치본에만 존재 REMOVED MEDIUM 운영본에만 존재 MISSING_REQUIRED CRITICAL 기준상 필수 파일이 한쪽에 없음 ERROR HIGH 비교 중 예외 발생 8. 파일 유형별 상세 비교 8.1 JAR 비교 • 1 단계: 존재 여부, 파일 크기, SHA-256 해시 비교 • 2 단계: zip 엔트리 목록 비교 - 추가/삭제/변경 엔트리 식별 • 3 단계: class, resource, META-INF 분리 집계 • 4 단계: 필요 시 public API 또는 디컴파일 비교 플러그인 연동 8.2 XML 비교 • 원문 해시 비교 후, 파싱 가능 여부 확인 • 공백/속성 순서/주석 제거 등 정규화 처리 • 노드 추가/삭제, 속성값 변경, 텍스트 값 변경 추적 • 주요 XPath 와 before/after 요약 제공

운영 권장안 실무 운영에서는 모든 JAR/XML 을 깊게 비교하기보다, 먼저 존재 여부와 해시를 확인하고 차이가 있는 파일만 상세 분 석하는 단계형 처리 방식이 효율적이다. 9. 화면 예상안 화면 핵심 구성 주요 목적 비교 실행 화면 경로 입력, 옵션 선택, 실행 버튼 비교 조건 설정 요약 대시보드 KPI 카드, 상태/유형/위험도 차트 전체 결과 분포 확인 비교 결과 목록 상태/유형/필수 필터, 검색, 상세 보기 변경 파일 탐색 상세 패널 해시, 엔트리, XPath, 메시지 개별 파일 검토 10. 리포트 산출물 산출물 구성 HTML 대시보드 요약 카드, 차트, 필터링 목록, 상세 패널 Excel SUMMARY 전체 건수, 상태별 차트, 유형별 차트 Excel DETAIL 파일별 전체 결과와 메시지 Excel JAR_DETAIL 추가/삭제/변경 엔트리, Manifest 변경 여부 Excel XML_DETAIL 변경 노드 수, 추가/삭제 노드, 주요 XPath 11. MVP 구축 범위 단계 포함 기능 비고 1 단계 기준 문서 로딩, 파일 스캔, 해시 비교, XML 정 규화 비교, JAR 엔트리 비교, Excel/HTML 리 포트 최소 실행 범위 2 단계 XPath 상세 diff, JAR class 수준 요약, 웹 대시 보드 고도화 운영 검토 생산성 향상 3 단계 JSON/YAML/DLL/WAR 확장, 승인 워크플로 우, CI/CD 연동 조직 표준화 12. 결론 본 설계안은 운영 배포본과 패치본의 차이를 단순 파일 비교 수준이 아니라, 기준 문서 준수 여부와 유형별 상세 검증까지 포함하는 검수 체계로 정리한 것이다. 구현 우선순위는 '기준 문서 -> 상대경로 병합 -> 상태 판정 -> HTML/XLSX 리포트' 흐름을 먼저 완성한 뒤, JAR/XML 상세 분석 과 웹 대시보드를 단계적으로 고도화하는 방향을 권장한다. 최종 제안 권장 초기 도입 형태: Python 비교 엔진 + baseline.xlsx + HTML/Excel 리포트. 이 구성은 개발 난이도와 운영 활용도의 균형이 가장 좋다.
